# Process

We're investigating a simple question: **can you pool K/V tensors before
attention computation without destroying the model, and if so, why?**

We don't know the answer. We have hunches. The hunches are usually wrong.
This document describes how we find out.

## Rules

1. **Every claim needs a measurement.** No "intuitively," no "likely," no
   "it's well known that." If you can't point to a number we measured, don't
   say it.

2. **Experiments are scripts, not notebooks.** Every experiment is a Python
   script in `experiments/`. It saves its output as JSON in `results/`. Anyone
   can re-run it and get the same numbers. Model revisions are pinned in the
   script.

   **NO ONE-OFF SCRIPTS.** Never run `python -c "..."` or `python3 -c "..."`
   for anything that produces a measurement. Every measurement must be a
   committed experiment script inheriting from `experiments.harness.Experiment`.
   The only exceptions are: `ls`, `cat`, `grep`, `git`, package installs,
   and 2-line import checks.

3. **Findings are atomic.** Each thing we learn goes in `findings/` as a short
   markdown file. One finding per file. The filename is the finding in
   sentence case, like `random-pooling-beats-skew-routing.md`. Each finding
   cites the experiment that produced it and the raw data file.

4. **We say when we're wrong.** Every finding has a `Status` field:
   `confirmed`, `uncertain`, `wrong`. When we discover something we believed
   is wrong, we update the finding to `wrong` with a note about when and why.
   We don't delete it. The record of being wrong is part of the science.

5. **Every experiment logs everything and generates figures.** Every experiment
   saves its full output as JSON in `results/` and generates at least one
   figure in `figures/`. No result lives only in a terminal.

6. **Test every registered model.** Experiments iterate `all_models()` from
   `experiments/models.py`. A model is only skipped when it fails to load
   (hardware/availability constraint) — failures are recorded in results, not
   silently ignored. No cherry-picking models to make results look better.

7. **Reproducibility before novelty.** Before we make a new claim, we verify
   the last one still holds. Re-run the confirming experiment. If the numbers
   drifted, investigate why before moving on.

## Workflow

```
Hunch → Hypothesis → Experiment → Data → Finding → (repeat)
```

1. **Hunch:** "I think X might be true."
2. **Hypothesis:** "If X is true, we should measure Y."
3. **Experiment:** Write a script that measures Y on real models with real
   data. It must have: a null hypothesis, a success criterion, a failure
   criterion. It must produce a JSON output file.
4. **Data:** Run it. Look at the numbers. Don't interpret yet — just record
   what came out.
5. **Finding:** Write `findings/<name>.md`. It has: what we measured, what
   the numbers were, what we conclude, what we're uncertain about, what
   experiment to run next. Status: `confirmed` or `uncertain`.

If the data contradicts the hunch, the finding says so. We update the hunch.
We don't massage the data. We don't run a different experiment that gives
the "right" answer. We follow the data.

## Agent Instructions

When asked to run an experiment, follow these steps in order. Do not skip steps.

**1. Identify the next experiment number.**
```bash
ls experiments/exp*.py | sort | tail -1
```

**2. Write the experiment script** using the template below. The module must be
named `expNN_<slug>.py`. Fill in all docstring fields before writing any code.

**3. Commit everything before running.**
The harness will refuse to run with uncommitted changes. Stage and commit the
new script (and any supporting changes to `rra/`) before proceeding.
```bash
git add experiments/expNN_<slug>.py
git commit -m "exp##: <one-line description>"
```

**4. Run the experiment.**
```bash
python -m experiments.expNN_<slug>
```
If it fails with a process violation, fix the violation and amend the commit,
then re-run. Re-runs are automatically numbered so no prior results are lost.

**5. Fill in the finding stub.**
`conclude()` writes `findings/expNN-<slug>.md` automatically. Open it and
fill in the `## What we measured`, `## What the numbers were`, and
`## What we're uncertain about` sections with the actual output.

**6. Commit results and finding.**
```bash
git add results/ figures/ findings/
git commit -m "exp##: results — <one-line summary of outcome>"
```

**7. Report back.**
State the verdict (confirmed / refuted), the key numbers, and what experiment
should come next. If the hypothesis was refuted, say so plainly.

## Experiment Template

Every experiment inherits from `experiments.harness.Experiment`. The harness
enforces these rules at runtime — missing any required call exits non-zero:

| Call | Required | What it does |
|------|----------|--------------|
| `expect(condition, label)` | yes (≥1) | Machine-checks a hypothesis criterion; verdict is derived from these |
| `save_results(data)` | yes | Saves JSON to `results/`, injects git hash, env, argv |
| `save_figure(fig, name)` | yes (≥1) | Saves PNG to `figures/` |
| `conclude(reason)` | yes | Records verdict derived from `expect()` calls; writes finding stub |

The harness also requires a clean git tree before running — commit your changes
first so the git hash stored in results is meaningful. If results already exist
for an experiment, the run is automatically numbered (`exp01_name_r2`,
`exp01_name_r3`, …) so no data is ever overwritten. The docstring is validated
at startup — missing required fields abort immediately.

```python
"""
Exp##: <one-line description>

Null hypothesis: <what would mean "this doesn't work">
Success:          <what we need to see to confirm it works>
Failure:          <what would disprove our hypothesis>

Model:     All registered models (experiments/models.py)
Data:      <dataset, split, size>
Hardware:  <machine specs>
Date:      <YYYY-MM-DD>
"""
from experiments.harness import Experiment


class Exp##Name(Experiment):
    def run(self):
        # ... do work ...

        self.expect(value > threshold, f"value={value:.3f} > {threshold}")  # required (≥1)
        self.save_results({"key": "value"})        # required
        self.save_figure(fig, "descriptive_name")  # required (at least once)
        self.conclude("one or two sentences citing key numbers")  # required


if __name__ == "__main__":
    Exp##Name().main()
```

Re-runs are automatically numbered: `exp01_name_r2.json`, `exp01_name_r3.json`, etc.
